# 05 · 코드 품질 검수 (Code Review)

> **목적**: 04 단계의 구현 결과를 **독립적인 시각**에서 점검하여, 사람 리뷰어에게 넘기기 전 명백한 결함을 잡는다.
> **선행 조건**: `prompts/output/04_implementation_log.md` 존재, 04에서 자체 검증 통과.
> **결과물**: `prompts/output/05_review_report.md`

---

## 역할 (Role)

너는 04 단계의 구현자와 **다른 사람**이다. 04의 결정을 옹호하지 않는다. 04에서 작성된 코드를 처음 보는 시니어 리뷰어처럼 의심하고 캐묻는다.

> 가능하면 이 단계는 별도 서브에이전트(`Agent` 도구의 `code-reviewer` 또는 `general-purpose`)로 실행한다. 같은 컨텍스트 안에서 자기 코드를 리뷰하면 확증 편향이 생긴다.

## 입력 (Inputs)

- `prompts/output/04_implementation_log.md` 의 "다음 단계 (05 검수)에 전달할 입력"
- `git diff main...HEAD` 또는 `git diff HEAD~N` 결과 (실제 변경된 라인)
- 03 설계 문서의 슬롯 4 (계약), 슬롯 6 (실패 시나리오)

## 검수 차원 (Review Dimensions)

다음 7개 차원 **각각**에서 "통과 / 우려 / 차단"을 판정한다. 차단이 1개라도 있으면 06 단계(테스트)로 진행하지 않고 04로 돌려보낸다.

### A. 설계 일치성 (Design Conformance)
- 04의 변경이 03의 계약(슬롯 4)과 일치하는가?
- 설계에 없던 변경이 들어가지 않았는가? (`git diff` 로 각 파일 점검)

### B. 외과적 변경 (Surgical Scope)
- 무관한 파일이 수정되었는가?
- 인접 코드 "개선" (스타일, 주석, 변수명)이 있는가?
- 사용처를 명확히 하지 않은 dead import / dead code 가 있는가?

### C. 보안 (Security)
- 시크릿 하드코딩 (`grep -E "(token|secret|key|password|api[_-]?key)" -i`)
- 입력 검증 누락 (특히 Telegram 메시지, 사용자 입력)
- 컨테이너: non-root 실행, 읽기 전용 파일시스템, 능력(capability) 최소화 유지되었는가?
- 의존성 신규 추가 시: 최근 CVE 알림 확인, 메인테이너 상태, 마지막 릴리스 시점

### D. 신뢰성 (Reliability)
- 예외 처리: 03 슬롯 6의 모든 실패 시나리오가 코드에 반영되었는가?
- 외부 호출 (Claude API, Telegram API) 에 타임아웃과 재시도 정책이 있는가?
- 비동기 흐름 (`asyncio`) 에 자원 누수가 없는가? (`async with`, `try/finally`)
- 로깅: 실패는 `logger.warning/error` 로 남기고 추적 가능한 컨텍스트 (작업 ID 등) 가 포함되는가?

### E. 가독성 / 유지보수성 (Readability)
- 함수가 한 가지 일을 하는가? (LoC 50 초과는 의심)
- 매직 넘버/문자열이 있는가?
- 명명이 일관적인가? (한 모듈에서 `task_id` 와 `taskId` 혼용 금지)
- 주석은 "무엇"이 아닌 "왜" 를 설명하는가?

### F. 테스트 (Test Quality)
- 변경된 행동마다 테스트가 있는가?
- 테스트가 mock 으로 가짜 통과를 만들지 않았는가? (특히 외부 API)
- 경계 조건 (빈 입력, 매우 큰 입력, 동시 호출) 이 다뤄지는가?
- 테스트가 결정적(deterministic)인가? (시간/순서/네트워크에 의존하지 않는가?)

### G. 운영성 (Operability)
- `docker compose up` 만으로 실행 가능한가?
- 환경 변수가 `.env.example` 또는 README 에 문서화되어 있는가?
- 헬스체크가 의미 있는 동작을 검사하는가? (단순히 `python --version` 같은 것은 불충분)
- 로그가 운영자가 알아볼 수 있는 형태인가? (구조화 로깅 권장)

## 자동 점검 명령 (Automated Checks)

직접 다음 명령을 실행하고 결과를 보고서에 포함한다. 실패한 명령은 우려/차단 사유로 기록한다.

```bash
# Python 코드 품질
uv run ruff check .
uv run ruff format --check .

# 타입 (도입되어 있다면)
# uv run ty check  또는  uv run mypy app/

# 의존성 보안 (도입되어 있다면)
# uv run pip-audit  또는  pip-audit

# Docker 빌드
docker build -t devops-pipeline:review .

# 컨테이너 보안 스캔 (도입되어 있다면)
# trivy image devops-pipeline:review

# SBOM 생성 (도입되어 있다면)
# syft devops-pipeline:review -o cyclonedx-json
```

도구가 설치되어 있지 않으면 "미설치 — 도입 권고" 로 기록한다.

## 출력 형식 (Output Format)

`prompts/output/05_review_report.md`:

```markdown
# 05 · 검수 보고서

## 0. 요약
- 차단(🔴): N건
- 우려(🟡): N건
- 통과(🟢): N건
- 다음 단계 진행 가능 여부: **예 / 아니오**

## 1. 차원별 판정
   ### A. 설계 일치성: 🟢/🟡/🔴 + 근거
   ### B. 외과적 변경: ...
   ### C. 보안: ...
   ### D. 신뢰성: ...
   ### E. 가독성: ...
   ### F. 테스트: ...
   ### G. 운영성: ...

## 2. 자동 점검 결과
   | 명령 | 결과 | 비고 |

## 3. 차단 항목 (반드시 04 로 돌아가서 수정)
   - 항목 1: [파일:라인] — [문제] — [수정 가이드]

## 4. 우려 항목 (06 단계 진행 가능하지만 추적 필요)
   - 항목 1: [파일:라인] — [문제] — [기록 위치 / 후속 티켓]

## 5. 06_test.md 로 넘길 입력
   - 추가 테스트 케이스 제안 (검수 중 발견한 빈틈)
   - 부하/성능 테스트 필요 여부
```

## 검증 기준 (Done Criteria)

- 7개 차원 모두에 판정과 근거가 있다.
- 차단이 있다면 04 단계로 명시적으로 돌려보낸다. "다음에 고치자"는 금지.
- 자동 점검 결과가 실제 명령 실행 결과로 포함된다 (가짜로 "통과" 적지 않는다).
- 모든 지적에 **파일:라인** 위치가 붙어 있다.

## 안티패턴 (절대 하지 말 것)

- 일반론적 권고만 적기 ("보안을 강화하세요").
- 04 단계의 결정을 옹호하기 위해 우려를 축소한다.
- 자동 점검을 건너뛰고 "코드만 봐도 괜찮다"고 결론낸다.
- 차단을 우려로 강등시켜 통과 처리한다.

## 인간 게이트 (Human Gate)

차단이 0건이고 우려가 사람이 받아들일 만한 수준이면, 다음 줄을 보고서 끝에 출력한다:

> ✅ **검수 통과**: 06 테스트 단계로 진행 가능. 우려 항목은 [후속 처리 위치] 에 기록됨.

차단이 있으면:

> ⛔ **차단**: 04 단계로 회귀 필요. 위 차단 항목을 모두 해결한 뒤 다시 05 를 실행하세요.

## 완료 알림 (Completion Notification)

검수 결과에 따라 둘 중 하나를 발송한다.

통과 시:

```bash
uv run python -m tools.notify "✅ *05 검수 통과* — 차단 0건, 우려 N건. 다음: 06 테스트"
```

차단 시:

```bash
uv run python -m tools.notify "⛔ *05 검수 차단* — 차단 N건. \`prompts/output/05_review_report.md\` 의 \"3. 차단 항목\" 확인 후 04 단계로 회귀 필요"
```
