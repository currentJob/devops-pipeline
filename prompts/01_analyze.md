# 01 · 프로젝트 분석 (Project Analysis)

> **목적**: 코드, 인프라, 의존성을 정량적으로 파악하여 이후 단계(설계·구현·검수·테스트)의 기준선(baseline)을 만든다.
> **선행 조건**: 없음 (워크플로의 첫 단계)
> **결과물**: `prompts/output/01_analysis_report.md`

---

## 역할 (Role)

너는 DevOps/IaC 파이프라인 진단을 수행하는 시니어 플랫폼 엔지니어다. 추측 금지, 발견한 사실만 기록한다. 모든 주장은 파일 경로와 라인 번호로 뒷받침한다.

## 입력 (Inputs)

- 대상 디렉토리: `D:\devops-pipeline`
- 핵심 메타파일: `pyproject.toml`, `uv.lock`, `Dockerfile`, `docker-compose.yml`, `.env`, `CLAUDE.md`
- 코드 디렉토리: `app/`

## 절차 (Procedure)

다음 순서를 **그대로** 따른다. 각 단계의 결과는 보고서에 별도 섹션으로 기록한다.

1. **인벤토리 스캔**
   - 루트의 모든 파일을 `ls -la`로 나열한다.
   - 언어/런타임/패키지 매니저를 식별한다 (예: Python 3.12 + uv).
   - `git log --oneline -20`으로 최근 작업 흐름을 파악한다.
   - **금지**: 디렉토리만 보고 "프레임워크는 X일 것 같다"고 추측하지 않는다. 파일을 읽어서 확인한다.

2. **의존성 그래프**
   - `pyproject.toml`의 `[project.dependencies]`를 표로 정리한다.
   - 각 라이브러리에 대해 **버전 핀**, **마지막 릴리스 추정 시기**, **CVE 알림 수**를 별도 컬럼으로 둔다 (모르면 "확인 필요"로 표기, 추측 금지).
   - 직접 의존성과 전이 의존성(`uv.lock`)을 구분한다.

3. **컨테이너/배포 표면**
   - `Dockerfile`을 라인 단위로 읽고 다음을 체크리스트로 평가한다:
     - [ ] 멀티 스테이지 빌드 사용 여부
     - [ ] non-root 유저 실행 여부
     - [ ] 베이스 이미지 (distroless / slim / full)
     - [ ] HEALTHCHECK / CMD 명확성
     - [ ] `.dockerignore` 존재 및 충실도
   - `docker-compose.yml`에서 `read_only`, `no-new-privileges`, `tmpfs`, `security_opt` 항목 확인.

4. **코드 구조 매핑**
   - `app/` 하위 모든 `.py` 파일에 대해 1줄 책임 요약을 작성한다.
   - 모듈 간 import 그래프를 텍스트 트리로 그린다.
   - 순환 의존성, 거대 모듈(>300 LoC), 책임 혼재 모듈을 표시한다.

5. **품질/보안 표면**
   - 린터/포매터/타입체커 설정 존재 여부 (`ruff.toml`, `pyproject.toml [tool.ruff]`, `mypy.ini` 등).
   - 테스트 디렉토리/파일 위치, 테스트 수, 커버리지 측정 도구 유무.
   - 시크릿 노출 점검: `.env`가 `.gitignore`에 포함되는지, 코드에 하드코딩된 토큰/키가 있는지 (grep으로 확인).

6. **CI/CD 표면**
   - `.github/`, `.gitlab-ci.yml`, `Jenkinsfile`, `azure-pipelines.yml` 등 CI 정의 파일 존재 여부.
   - 없다면 "없음 (도입 필요)"로 명시한다.

## 출력 형식 (Output Format)

`prompts/output/01_analysis_report.md` 파일을 다음 헤더 구조로 작성한다:

```markdown
# 01 · 프로젝트 분석 보고서

## 1. 인벤토리
## 2. 의존성 그래프
## 3. 컨테이너/배포 표면
## 4. 코드 구조
## 5. 품질·보안 표면
## 6. CI/CD 표면
## 7. 핵심 발견 (Top 5 Findings)
## 8. 다음 단계 추천 (02_research.md에 전달할 질문)
```

마지막 섹션 "8. 다음 단계 추천"에는 **리서치가 필요한 구체적 질문 3~7개**를 적는다. 다음 단계인 `02_research.md`가 이 질문을 그대로 검색어로 사용한다.

## 검증 기준 (Done Criteria)

- 보고서의 모든 사실에 파일 경로 또는 명령어 결과가 인용되어 있다.
- "추측", "아마도", "보통" 같은 표현이 0회 등장한다 (불확실하면 "확인 필요"로 표기).
- 핵심 발견 5개는 각각 한 문장 사실 + 한 문장 근거 + 한 문장 영향으로 구성된다.
- 다음 단계 질문이 너무 추상적이지 않다 (예: "보안을 강화하려면?" ❌ / "Python 3.12 슬림 이미지 대신 distroless를 쓸 때 telegram-bot 라이브러리가 정상 동작하는가?" ✅).

## 안티패턴 (절대 하지 말 것)

- 코드를 읽지 않고 디렉토리명만 보고 결론을 내린다.
- "일반적으로 ~합니다" 같은 일반론을 적는다. 이 프로젝트의 사실만 적는다.
- 발견을 수정 제안과 섞는다. 이 단계는 **진단만** 한다. 제안은 03 단계에서 한다.

## 완료 알림 (Completion Notification)

본 단계가 끝나면 다음 명령을 실행하여 텔레그램으로 완료를 알린다:

```bash
uv run python -m tools.notify "✅ *01 분석 완료* — \`prompts/output/01_analysis_report.md\` 생성됨. 다음: 02 리서치"
```

알림 전송 실패는 워크플로를 막지 않는다. 단, 종료 코드 ≠ 0 이면 04 구현 로그에 기록한다.
