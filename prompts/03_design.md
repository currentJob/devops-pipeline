# 03 · 아키텍처 설계 검증 (Architecture Design)

> **목적**: 02 단계의 트렌드 권고를 이 프로젝트의 실제 변경안으로 번역하고, 대안 비교·트레이드오프·실패 시나리오까지 검토한 설계 문서를 만든다.
> **선행 조건**: `prompts/output/02_research_brief.md` 가 존재해야 한다.
> **결과물**: `prompts/output/03_design_doc.md`

---

## 역할 (Role)

너는 시니어 소프트웨어/플랫폼 아키텍트다. 설계는 **항상 최소 2개의 대안**을 비교한 뒤 선택한다. "이게 정답"이라는 단정 금지, "X 조건에서는 A가, Y 조건에서는 B가 낫다"는 식의 조건부 권고만 허용된다.

## 입력 (Inputs)

- `prompts/output/02_research_brief.md` 의 "4. 03_design.md 로 넘길 입력" 섹션
- 프로젝트 현재 구조 (`app/main.py`, `app/pipeline.py`, `app/notifier.py`, `app/commands.py`, `app/config.py`)
- 비기능 요구사항: 1인 운영, Telegram 승인 흐름 유지, Claude API 호출, Docker 단일 컨테이너

## 절차 (Procedure)

각 권고(02 브리프의 상위 5개)에 대해 다음 7개 슬롯을 **빠짐없이** 채운다.

### 슬롯 구조

```markdown
### 권고 N: [한 줄 제목]

#### 1. 변경 범위 (Scope)
- 영향받는 파일: [경로 + 라인 범위]
- 영향받는 외부 시스템: [Docker 이미지, CI, 모니터링 등]
- 변경 LoC 추정: [숫자]

#### 2. 대안 (Alternatives)
| 대안 | 설명 | 장점 | 단점 |
|---|---|---|---|
| A | ... | ... | ... |
| B | ... | ... | ... |
| C (현상유지) | 변경 안 함 | ... | ... |

#### 3. 선택 (Selection)
- 선택: **대안 [X]**
- 이유: [3문장 이내, 비기능 요구사항과 연결]

#### 4. 인터페이스/계약 (Contracts)
- 변경되는 함수/모듈 시그니처
- 환경 변수 추가/제거/이름 변경
- Docker 빌드 인자 변경
- 외부 API 의존성 변경

#### 5. 마이그레이션 단계 (Migration Steps)
1. [단계] → 검증: [어떻게 확인?]
2. [단계] → 검증: [어떻게 확인?]
3. [단계] → 검증: [어떻게 확인?]

#### 6. 실패 시나리오 (Failure Modes)
- 시나리오 1: [실패 조건] → 영향: [무엇이 깨지나] → 복구: [어떻게 되돌리나]
- 시나리오 2: ...

#### 7. 의사결정 트리거 (Reversibility)
- 이 결정을 되돌려야 할 조건: [구체적 임계값]
- 되돌리는 비용 추정: [낮음/중간/높음] + 이유
```

## 횡단 점검 (Cross-cutting Checks)

개별 권고를 다 채운 후, 마지막 섹션에서 다음을 점검한다.

1. **충돌 검사**: 권고 간 충돌이 있는가? (예: "distroless로 전환"과 "헬스체크에 `python` 실행"이 충돌할 수 있음)
2. **순서 의존성**: 어느 권고가 먼저 들어가야 하는가? DAG로 그린다.
3. **공유 자원**: 같은 파일을 동시에 수정하는 권고들이 있는가? 머지 충돌 위험.
4. **롤백 일관성**: 모든 권고가 독립적으로 롤백 가능한가? 그렇지 않다면 묶음 단위를 정의한다.

## 출력 형식 (Output Format)

`prompts/output/03_design_doc.md`:

```markdown
# 03 · 아키텍처 설계 문서

## 0. 의사결정 요약 (Decision Log)
| # | 권고 제목 | 선택 대안 | 변경 LoC | 위험도 |

## 1. 권고별 설계 (위 슬롯 구조)
   ### 권고 1: ...
   ### 권고 2: ...
   ...

## 2. 횡단 점검
   ### 2.1 권고 간 충돌
   ### 2.2 적용 순서 (DAG)
   ### 2.3 공유 자원 / 머지 위험
   ### 2.4 롤백 단위

## 3. 04_implement.md 로 넘길 작업 큐
   - [ ] 작업 1: [한 줄 + 추정 시간]
   - [ ] 작업 2: ...
```

## 검증 기준 (Done Criteria)

- 각 권고마다 7개 슬롯이 빠짐없이 채워져 있다.
- 대안 비교에서 "현상유지" 옵션이 항상 포함되어 있다.
- 실패 시나리오가 최소 2개 이상이며, 각각 복구 방법이 명시되어 있다.
- 횡단 점검 4개 항목이 모두 답변되어 있다.
- 작업 큐의 각 항목이 1시간 이내 단위로 쪼개져 있다 (큰 작업은 더 쪼갠다).

## 안티패턴 (절대 하지 말 것)

- "베스트 프랙티스이므로 도입한다" → 비기능 요구사항과의 연결 없이 권위에 호소.
- 대안을 1개만 적고 선택한다.
- "실패하면 롤백한다" 같은 공허한 복구 계획 → 어떤 명령/단계로 롤백할지 적어야 한다.
- 설계 단계에서 코드를 쓴다 → 코드는 04 단계의 일이다. 여기는 시그니처와 계약만.

## 인간 게이트 (Human Gate)

이 단계의 결과물은 **반드시 사람의 검토를 거쳐야** 다음 단계로 진행한다. Telegram 승인 흐름과 마찬가지로, 자동 진행 금지. 보고서 끝에 다음 줄을 출력한다:

> ⏸️  **승인 대기**: 위 설계를 검토하고 "04 진행" 또는 "수정 필요: [내용]" 으로 응답해주세요.

설계 문서 작성이 끝나면(승인 대기 진입 시점) 다음을 실행한다. 사람이 즉시 검토할 수 있도록 알린다:

```bash
uv run python -m cli.notify "⏸️ *03 설계 완료 — 승인 대기* \`prompts/output/03_design_doc.md\` 검토 후 \"04 진행\" 으로 응답해주세요"
```
