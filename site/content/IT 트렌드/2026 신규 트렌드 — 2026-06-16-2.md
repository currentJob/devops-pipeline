## WebGPU — 브라우저 내 ML/GPU 추론

### 2026-06-16 기준 트렌드 개요
WebGPU는 2025년 주요 브라우저 기본 활성화 이후, 2026년 5~6월 들어 **llama.cpp WebGPU 백엔드**, **RL 에이전트 브라우저 실행**, **물리 시뮬레이션** 등 실용 프로젝트가 HN 상위에 연속 등장하며 성숙기로 진입하고 있다. 특히 `llama.cpp` WebGPU 지원([HN 2026-05-21](https://reeselevine.github.io/llamas-on-the-web/))과 [wasi-webgpu GitHub proposal(2026-06-12)](https://github.com/WebAssembly/wasi-webgpu)은 서버리스·엣지 환경까지 GPU 추론을 확장하는 핵심 지점이다. 채택률은 '실험적 프로덕션' 단계로, Chrome/Firefox 모두 플래그 없이 사용 가능하나 모바일 지원은 아직 제한적이다.

### 핵심 기술 스택
- **API**: WebGPU API (W3C 표준), WGSL(셰이더 언어)
- **런타임**: `llama.cpp` WebGPU 백엔드, `ggml` WebGPU, `transformers.js` (WebGPU 옵션)
- **프레임워크**: `wgpu` (Rust), `dawn` (C++), `webgpu.rocks` 샘플
- **연계**: WASI-WebGPU (WebAssembly System Interface 확장)

### 적합한 역할
- 프론트엔드 엔지니어 (브라우저 GPU 활용)
- ML 엔지니어 (온디바이스 추론 최적화)
- 게임/시뮬레이션 개발자

## Local-First AI 메모리 & 데이터

### 2026-06-16 기준 트렌드 개요
"Local-First"는 CRDT·Automerge·Yjs 기반 협업 데이터를 넘어, 2026년에는 **AI 에이전트의 로컬 메모리 레이어**로 확장되고 있다. HN에서 [Mnemo(2026-06-03)](https://github.com/zaydmulani09/mnemo), [Cortex 암호화 에이전트 메모리(2026-06-13)](https://github.com/gambletan/cortex), [Mutualism Accord 프로토콜(2026-06-13)](https://mutualismaccord.org), [Eatmydata.ai Local-First SQL(2026-06-10)](https://eatmydata.ai/)이 잇따라 등장했다. 서버에 의존하지 않고 사용자 장치에서 데이터 주권을 유지하면서 AI를 활용하려는 수요가 실제 프로젝트로 구체화되는 초기 성장 단계다.

### 핵심 기술 스택
- **CRDT 라이브러리**: `Automerge`, `Yjs`, `diamond-types`
- **로컬 스토리지**: SQLite (Rust/WASM), DuckDB (인-프로세스), `libsql`
- **AI 메모리**: `Mnemo` (Rust + SQLite + petgraph), `Cortex` (Rust + MCP)
- **동기화**: Electric SQL, PowerSync, `replicache`
- **보안**: 로컬 암호화(AES-GCM), E2E 키 관리

### 추천 학습 자료
- [Local-First Software 원문 (Ink & Switch)](https://www.inkandswitch.com/local-first/)
- [Automerge 공식 문서](https://automerge.org/)
- [Yjs 문서](https://docs.yjs.dev/)
- [Mnemo GitHub](https://github.com/zaydmulani09/mnemo)
- [Cortex GitHub](https://github.com/gambletan/cortex)

### 적합한 역할
- 풀스택/앱 개발자 (오프라인 우선 앱)
- AI 에이전트 개발자 (지속 메모리 설계)
- 보안·프라이버시 엔지니어

## MCP (Model Context Protocol) 생태계

### 2026-06-16 기준 트렌드 개요
Anthropic이 제안한 **Model Context Protocol(MCP)** 은 AI 에이전트와 외부 툴·데이터소스를 연결하는 JSON-RPC 기반 표준으로, 2026년 6월 현재 Ruby, Go, Python 등 다양한 언어 클라이언트가 커뮤니티 주도로 등장하고 있다. 특히 [Lumen — Binary Alternative to JSON-RPC for MCP(2026-06-15)](https://github.com/GonzaloMonzonC/lumen-protocol/)는 MCP의 성능 한계를 극복하려는 시도로 HN 최상단에 올랐으며, 이는 MCP가 '실험'을 넘어 **프로덕션 최적화 단계**로 진입했음을 보여준다. Vault 내 기존 AI 에이전트 설계 노트가 MCP를 언급하나 생태계 독립 카테고리로 다루지는 않았다.

> [!tip] 이 프로젝트와 직접 연관
> 현재 DevOps 파이프라인의 `vault_*` 도구 설계가 MCP 아키텍처와 구조적으로 유사하다. MCP 표준 채택 시 에이전트 툴 인터페이스를 표준화할 수 있다.

### 핵심 기술 스택
- **표준**: MCP Spec (JSON-RPC 2.0 기반), SSE/WebSocket 전송
- **공식 SDK**: `@anthropic-ai/sdk` (TypeScript), `anthropic` (Python)
- **커뮤니티**: `manceps` (Ruby), `lumen-protocol` (Binary 대안)
- **서버 예시**: MCP Filesystem Server, GitHub MCP Server, Postgres MCP Server
- **연계**: Claude Desktop, Cursor, 커스텀 에이전트 워커

### 추천 학습 자료
- [MCP 공식 문서](https://modelcontextprotocol.io/)
- [MCP 스펙 GitHub](https://github.com/modelcontextprotocol/specification)
- [Lumen Protocol GitHub](https://github.com/GonzaloMonzonC/lumen-protocol/)
- [Anthropic MCP 소개 블로그](https://www.anthropic.com/news/model-context-protocol)

### 적합한 역할
- AI 에이전트 개발자
- 백엔드/플랫폼 엔지니어 (툴 서버 구축)
- DevOps 엔지니어 (인프라 연동 MCP 서버)

## LLM Observability — OpenTelemetry × LLMOps

### 2026-06-16 기준 트렌드 개요
LLM 앱의 프로덕션 운영이 보편화되면서 **OpenTelemetry(OTel)를 LLM 파이프라인에 적용**하는 수요가 급증하고 있다. [Observing LLM Applications with OpenTelemetry(2026-06-05)](https://signoz.io/blog/opentelemetry-llm/)와 [Claude Code 품질 저하 측정(2026-05-26)](https://signoz.io/blog/claude-code-measure-degradation-opentelemetry/)이 HN에서 주목받았고, [Superlog YC P26(2026-05-19)](https://superlog.sh/)처럼 자동 설치형 LLM 관측 스타트업도 등장했다. 성숙도는 '초기 표준화' 단계로, OTel의 GenAI 시맨틱 컨벤션이 2026년 초 Draft에서 RC(Release Candidate)로 승격됐다.

> [!warning] 주의
> OTel GenAI 시맨틱 컨벤션은 아직 안정 릴리즈가 아니므로 속성명이 변경될 수 있다. 프로덕션 도입 시 버전 고정 필수.

### 핵심 기술 스택
- **계측**: OpenTelemetry SDK (Python/TypeScript), `opentelemetry-instrumentation-anthropic`
- **백엔드**: SigNoz, Jaeger, Grafana Tempo, Honeycomb
- **메트릭**: 토큰 사용량, 레이턴시 분포, 에러율, 모델별 비용
- **표준**: OTel GenAI Semantic Conventions (Draft → RC)
- **특화 도구**: LangFuse, Helicone, Braintrust (LLM 전용 관측)

### 추천 학습 자료
- [OpenTelemetry 공식](https://opentelemetry.io/)
- [Observing LLM Apps with OTel — SigNoz](https://signoz.io/blog/opentelemetry-llm/)
- [OTel GenAI Semantic Conventions](https://opentelemetry.io/docs/specs/semconv/gen-ai/)
- [LangFuse 문서](https://langfuse.com/docs)
- [Superlog (YC P26)](https://superlog.sh/)

### 적합한 역할
- ML 엔지니어 / LLMOps 엔지니어
- DevOps·SRE (AI 인프라 운영)
- 백엔드 엔지니어 (LLM 앱 개발)

## WASI-WebGPU — WebAssembly × GPU 시스템 인터페이스

### 2026-06-16 기준 트렌드 개요
**WASI-WebGPU**는 WebAssembly 모듈이 플랫폼 독립적으로 GPU 자원에 접근할 수 있도록 하는 WASI(WebAssembly System Interface) 확장 제안이다. [wasi-gfx 미래 로드맵 포스트(2026-06-09)](https://wasi-gfx.dev/blog/posts/future-of-wasi-gfx/)와 [WebAssembly/wasi-webgpu GitHub 공식 제안(2026-06-12)](https://github.com/WebAssembly/wasi-webgpu)이 6월 둘째 주에 연속 HN 상위에 올라 커뮤니티의 높은 관심을 확인했다. WebGPU(브라우저)와는 달리 서버·엣지·임베디드 환경에서의 GPU 접근이 목표로, WebAssembly 컴포넌트 모델과 결합해 이식성 높은 AI/그래픽 런타임을 가능케 한다. 현재 초기 제안(Proposal) 단계로 성숙도는 낮지만 표준화 속도가 빠르다.

> [!warning] 주의
> WASI-WebGPU는 2026-06-16 기준 아직 W3C/WASI 표준 확정 전 단계다. 프로덕션 활용보다는 선행 학습·기여 관점에서 접근 권장.

### 핵심 기술 스택
- **표준**: WASI Preview 2 (컴포넌트 모델), `wasi-webgpu` Proposal
- **런타임**: `wasmtime` (GPU 확장), `WasmEdge` (WASI-NN/GPU)
- **언어**: Rust (`wasm-bindgen`, `wgpu`), C/C++ (`wasi-sdk`)
- **연계**: Cloudflare Workers (WASM), Fastly Compute@Edge

### 추천 학습 자료
- [wasi-webgpu GitHub Proposal](https://github.com/WebAssembly/wasi-webgpu)
- [wasi-gfx 블로그](https://wasi-gfx.dev/blog/posts/future-of-wasi-gfx/)
- [WASI 공식 문서](https://wasi.dev/)
- [Wasmtime 런타임](https://wasmtime.dev/)

### 적합한 역할
- 시스템 프로그래머 (Rust/C++)
- WebAssembly/엣지 컴퓨팅 엔지니어
- 표준화·오픈소스 기여자
